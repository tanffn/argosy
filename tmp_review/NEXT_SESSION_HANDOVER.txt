PART 1 — THE AGENDA: build the PROPOSALS surface (the daily proactive agent)
********************************************************************************

The four-surface product model (now in SDD §11.0 — read it):
  1. PLAN       — strategic planning / forecasts / Monte Carlo (generates THE PLAN).
  2. RETIREMENT — tracking + expected date (OWNS the FI projection; §19).
  3. PORTFOLIO  — current vs target + free cash USD&NIS + what-we-own.  ← DONE this session.
  4. PROPOSALS  — actions to take.  ← THIS SESSION'S BUILD.

PROPOSALS' one job: surface (a) how to DEPLOY free cash and (b) NEW buy
opportunities and hold/sell/buy calls — DRIVEN BY A DAILY AGENT that reacts to
BOTH a free-cash change AND per-stock market-sentiment changes. Broad/long ETFs
rarely move; INDIVIDUAL STOCKS might — so the daily "do we need to act?" check
must reason at the per-holding / per-candidate level, not just on cash deltas.

The pieces EXIST but are NOT assembled into one daily agent (from a prior survey):
  - argosy/agents/state_observer.py + orchestrator/loops/state_observer.py
      runs daily ~17:00 IDT, writes MonitorFlag rows (emergent anomaly flagger).
  - argosy/agents/action_proposer.py + services/action_proposer_runner.py
      turns a trigger (flag / snapshot / inferred-event) into 0-3 ActionProposal
      rows. NEVER executes. BUT nothing auto-fires it daily off the flags.
  - argosy/agents/sentiment_analyst.py — per-ticker bullish/neutral/bearish.
      EXISTS but standalone: no scheduled run, no data feed wired. (The
      per-stock sentiment signal is the genuinely-new dependency to source.)
  - GET /api/portfolio/deploy-cash (services/deployment_advisor.py) — tiered,
      estate-aware plan to deploy idle cash toward the plan target. Ready.
  - orchestrator/loops/watchlist.py — daily; emits a WS event; doesn't drive proposals.
  - UI: ui/src/app/proposals/page.tsx — ActionProposalCard + DeployCashCard +
      Windfall/TrendRadar/SpeculativeMonitor/Discovery cards already render.
  - Routes: /api/proposals/actions* (action proposals) + /api/proposals (orders).

THE BUILD = one daily orchestration loop ("do we need to act?") that:
  (a) reads current cash (USD+NIS — now first-class, see PART 2) + holdings,
  (b) detects per-holding market-sentiment shifts (wire the sentiment analyst +
      its data feed — the new piece; broad ETFs skip, individual names reasoned),
  (c) ties today's state_observer flags → action_proposer (currently un-wired),
  (d) emits proposals: deploy-cash plans (reuse deploy-cash), new buys, hold/sell/buy.
Connect to the EXISTING Deployment Advisor + alternatives sourcing rather than
building parallel logic. SCOPE the daily agent's cadence / "what counts as act"
with Ariel before building (front-load the judgment calls, then sprint).
