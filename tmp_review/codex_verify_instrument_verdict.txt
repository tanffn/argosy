1. **BLOCKER**: Partial registry rows can verify agent-supplied facts.
[instrument_verification.py](D:/Projects/financial-advisor/argosy/services/instrument_verification.py:133) falls back to `claimed_isin` / `claimed_domicile` on a registry hit. With registry `FAKE: {}`, this returns `verified=True`, `GREEN`:

```python
verify_instrument(symbol="FAKE", claimed_domicile="IE", claimed_isin="IE00B579F325")
```

Fix: never fall back to claims after a registry hit; require schema-valid `isin`, `domicile`, `source_url`, `verified_on`, or fail closed.

2. **BLOCKER**: US domicile is only detected when exactly `"US"`.
[instrument_verification.py](D:/Projects/financial-advisor/argosy/services/instrument_verification.py:138) does not `strip()` or normalize synonyms. Registry row:

```yaml
USFAKE:
  isin: IE00B579F325
  domicile: United States
```

then:

```python
verify_instrument(symbol="USFAKE", claimed_domicile="IE", claimed_isin="IE00B579F325")
```

returns `verified=True`, `GREEN`. Exact `domicile: US` is rejected, but non-canonical US spellings bypass.

Fix: normalize domicile through a closed enum/synonym map and reject unknown registry domicile values.

3. **HIGH**: A known symbol with contradictory claimed ISIN/domicile returns GREEN, but the canonical registry facts are not returned.
Current registry input:

```python
verify_instrument(symbol="SGLD", claimed_domicile="US", claimed_isin="US4642875235")
```

returns `verified=True`, `GREEN` using registry SGLD facts. That is only safe if downstream discards the claimed fields, but `VerificationResult` / `VerificationEvidence` do not expose authoritative `isin` or `domicile`, and `VerifiedAlternativesCandidate` stores independent `domicile` / `isin` without validating them against the verification result: [alternatives_types.py](D:/Projects/financial-advisor/argosy/services/alternatives_types.py:40).

Fix: return canonical verified `isin` and `domicile`, and require candidates to use those exact facts or reject claim/registry mismatches.

4. **MEDIUM**: Registry-confirmed instruments are wrongly rejected on trivial symbol formatting.
`registry_lookup()` uppercases but does not strip: [instrument_verification.py](D:/Projects/financial-advisor/argosy/services/instrument_verification.py:104). This clean current-registry input is rejected as `YELLOW`:

```python
verify_instrument(symbol=" sgld ", claimed_domicile="IE", claimed_isin="IE00B579F325")
```

Fix: canonicalize symbols with `strip().upper()` and consider explicit listing aliases such as `SGLD.L` / `SGLD LN`.

5. **MEDIUM**: The ISIN prefix allowlist rejects valid non-US ISO prefixes before registry facts can help.
[instrument_verification.py](D:/Projects/financial-advisor/argosy/services/instrument_verification.py:29) omits many valid ISO prefixes. Example `SE0000108656` has a valid ISO checksum, but `isin_is_valid()` returns `False`, so a registry-confirmed non-US Swedish instrument would be `RED`.

Fix: separate checksum validity from book policy; use a complete ISO/NNA prefix set, then apply estate policy via normalized registry domicile.

Edge checks: `None` / empty ISIN do not return GREEN; mixed-case ISIN works; `XS` works; exact `domicile="US"` with non-US prefix rejects. I could not run pytest because `pytest` is not installed in this environment, but I did validate the concrete examples with direct Python probes.
