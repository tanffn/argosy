## 1. Purpose

EX1 landed 4 working parsers, ingest pipeline, REST/CLI surface, and the `HouseholdCategorizerAgent` — but the SDD handover catalogued seven defects (some real-bug, some cosmetic) and left operational items hanging (stale uvicorn, partial backfill). Wave EX1.1 closes that list, then runs a fresh code re-read + corpus aggregate smoke to surface anything else the autonomous EX1 implementation missed. End-state: the ingest is correct enough to build wave-2 verification and wave-3 dashboard on top of without re-doing it.

## 2. Scope

### In scope

- 7 named bug-fixes (Section 4).
- New top-level `argosy/services/fx/` module with BoI-cached daily rates (Section 5). Used by future dashboard, not by parsers.
- Migration `0022_expense_amount_nis_nullable` + `0023_fx_rates`.
- Operational reset: stop services → migrate → wipe per-user expense tables → full corpus re-ingest → restart services.
- Verify-phase code re-read + 8 corpus-aggregate SQL exercises; deliverable is `docs/superpowers/specs/2026-05-09-ex1.1-verify-findings.md`.
- ~14 new tests (7 per-bug regressions + 5 FX unit tests + 1 corpus-replay integration + 1 endpoint test).

### Out of scope (explicitly)

- **Per-row data verification with the user** — wave 2.
- **Dashboard FX toggle / visualizations** — wave 3.
- **Multi-tenant Leumi account-extraction** — bug #3 fix is a *guard*, not a multi-tenant rewrite.
- **EX2 anomaly detection** (Card 2923 fee-waiver flag, novel-merchant alerts) — separate wave.
- **Cal/Amex/Diners parsers** — still `NotImplementedError` stubs.

## 4. Bug-fix detail

### Bug 1 — Max parser learns card last-4 from caller

**Files:** `argosy/services/expense_ingest/parsers/max.py`, `argosy/services/expense_ingest/orchestrator.py`, `argosy/cli/expenses.py`, `argosy/api/routes/expenses.py`.

**Change:**

- `p_max.parse(path, *, last4_hint: str | None = None) -> ParseResult`. If `last4_hint` provided, `source_hint.external_id = last4_hint`. Else fall back to today's behavior (extract from sheet name) and emit a `warnings.warn("Max parser: no last4_hint, falling back to bank-account last-4 — likely wrong")`.
- Orchestrator threads `last4_hint` through to `parser_fn(path, last4_hint=...)`. The hint is sourced from `UserFile.metadata['card_last4']` if present.
- CLI backfill walker (`argosy expenses backfill`) extracts last-4 from the parent folder name when ingesting a file under `Cards/Max/<last4>/<file>.xlsx`. Stores it on the `UserFile.metadata` row at catalog time.
- REST upload route accepts an optional `card_last4` form field. If parser detection returns Max issuer and `card_last4` is missing from the form, return `400 Bad Request` with body `{"detail": "card_last4 required for Max uploads"}`. If present, store on `UserFile.metadata` before kicking ingest.

**Why folder-name + form field, not filename:** Existing files aren't named with the last-4 (e.g. `Apr.xlsx`, not `max-6225-Apr.xlsx`); renaming ~17 files is friction we avoid. Folder name is already the convention you use; making it load-bearing is cheap.

**Decision:** API returns 400 (rather than ingest with placeholder + PATCH later) because it surfaces the missing data early. User sign-off recorded 2026-05-09.

### Bug 2 — Foreign `amount_nis` becomes NULL

**Files:** `argosy/services/expense_ingest/parsers/isracard.py`, `argosy/state/models.py`, `argosy/services/expense_ingest/correlator.py`, `argosy/services/expense_ingest/refund_matcher.py`, `argosy/api/routes/expenses.py` (monthly-summary), Migration `0022`.

**Change:**

- Isracard parser: replace lines 152–155 so `amount_nis = None` when `charge_ccy != "NIS"`. `amount_orig` and `currency_orig` already populated (lines 180–181).
- `state/models.py`: `ExpenseTransaction.amount_nis: Mapped[float | None]`. Update all type hints.
- Migration `0022_expense_amount_nis_nullable`: ALTER COLUMN to drop `NOT NULL`. SQLite quirk: requires table-rebuild migration (CREATE TABLE new + INSERT SELECT + DROP + RENAME). Standard alembic batch operation.
- Correlator: when matching Leumi card-payment lines against statement totals, sum only NIS rows (`WHERE amount_nis IS NOT NULL`). Document the choice — the Leumi line is denominated in NIS, so the bank's match is against NIS-charged sum only.
- Refund matcher: when matching a credit row to a prior debit, if either side is NULL `amount_nis`, fall back to matching on `amount_orig + currency_orig`.
- Monthly-summary endpoint returns `{"NIS": 12345.67, "USD": 250.00, "EUR": 0.0}` per-currency map; dashboard handles display (wave 3).
- Conservation tests already only sum NIS-charged rows (parser line 172–173) — no change needed.

**Why NULL not converted-on-ingest:** Per Q4 / user sign-off — store what actually happened (foreign amount in original currency); convert on demand at display time using the FX module. Avoids committing to an FX rate at ingest that may not match what the bank quietly used.

### Bug 3 — Leumi account-number guard

**Files:** `argosy/services/expense_ingest/parsers/leumi_osh.py`, `argosy/services/expense_ingest/orchestrator.py`.

**Change:**

- Leumi parser extracts the account number from the HTML header. Exact selector confirmed during impl (Leumi statements are HTML; account number lives in a `<title>` or `<header>` element).
- Parser stores extracted account number on `ParseResult.source_hint.external_id` (parser now fills `source_hint`, no longer relying on orchestrator default).
- `orchestrator._leumi_source_hint` is renamed to `_leumi_source_hint_assert`. It receives the parsed `ParseResult`, asserts the parsed account number matches the hardcoded `"44745280"`, and raises `ValueError("Leumi account mismatch: expected 44745280, got <X>")` if not.
- Hardcoded value stays (single-user simplification). The assert is the cheap multi-tenant trip-wire.

**Why guard not full extraction:** Per Q6 / user sign-off — single-user, no observable benefit from removing the hardcode, but a cheap "blow up if surprised" check is worth having.

### Bug 4 — N+1 query in category resolver

**Files:** `argosy/services/expense_ingest/category_resolver.py`.

**Change:** Replace per-tx `session.get(ExpenseSource, tx.source_id)` with a single batch query at the start of the resolver function: collect all `source_id`s, fetch as one `WHERE id IN (...)` query, build a dict, look up per tx. Saves N-1 round-trips per LLM batch.

### Bug 5 — Model alias resolution

**Files:** `argosy/agents/base.py`.

**Change:** `DEFAULT_MODEL_BY_ROLE["household_categorizer"] = "claude-sonnet-4-6"` (was `"sonnet"`). Aligns with how every other agent role declares its model in this dict.

### Bug 6 — `categories_resolved` counter overcount

**Files:** `argosy/services/expense_ingest/category_resolver.py`.

**Change:** Move the `resolved += 1` increment inside the branch that actually assigns a category to a tx, after checking the LLM result for `"uncategorized"`. Today the increment fires before the check, so rows that ended up uncategorized are still counted. Fix is a single line move.

### Bug 7 — Leumi `raw_row` semantic keys

**Files:** `argosy/services/expense_ingest/parsers/leumi_osh.py`.

**Change:** Replace integer-string keys (`"0"…"8"`) with semantic keys: `{"date", "value_date", "description", "reference", "debit", "credit", "balance", ...}`. Exact key set confirmed during impl by reading Leumi HTML column headers. Matches the convention used by `max.py`, `isracard.py`, `discount.py`. Existing rows in DB are wiped and re-ingested by the operational reset (Section 6), so no in-place data migration needed.

## 5. New `fx` module

**Path:** `argosy/services/fx/__init__.py` (top-level — likely useful for portfolio NAV in NIS, plan synthesis, etc.).

### 5.1 Public API

```python
from datetime import date

def convert(amount: float, from_ccy: str, to_ccy: str, on: date) -> float:
    """Convert amount from from_ccy to to_ccy on the given date."""

def rate(from_ccy: str, to_ccy: str, on: date) -> float:
    """Return the exchange rate (units of to_ccy per 1 from_ccy) on on."""

def warm_cache(start: date, end: date, currencies: list[str]) -> int:
    """Bulk-prefetch daily rates from BoI for the given date range. Returns rows fetched."""
```

### 5.2 Behavior

- ISO 4217 codes (`USD`, `EUR`, `GBP`, `ILS`). `"NIS"` is normalized to `"ILS"` on entry.
- Same-currency: short-circuit, return amount unchanged.
- Cross-rates (`USD → EUR`): two-hop via ILS — `convert(amount, "USD", "ILS", on) → "EUR"`.
- Missing date (weekend / holiday): walk back day-by-day up to 7 days. If still none, raise `FXRateUnavailable`.
- BoI API failure with cache miss: raise `FXRateUnavailable`. Do **not** silently fall back to a static constant — that's the bug we're fixing.

### 5.3 Storage

Migration `0023_fx_rates`:

```sql
CREATE TABLE fx_rates (
    date         DATE   NOT NULL,
    currency     TEXT   NOT NULL,
    rate         REAL   NOT NULL,           -- units of ILS per 1 unit of currency
    source       TEXT   NOT NULL DEFAULT 'boi',
    fetched_at   TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    PRIMARY KEY (date, currency)
);
CREATE INDEX idx_fx_rates_currency ON fx_rates(currency, date);
```

All rates stored against ILS; cross-rates derived.

### 5.4 BoI source

- Endpoint: `https://www.boi.org.il/PublicApi/GetExchangeRates?startDate=YYYY-MM-DD&endDate=YYYY-MM-DD&currencies=USD,EUR,GBP`. **Exact URL to be re-confirmed during implementation** — BoI changed their API surface in 2023.
- HTTP client: `httpx` (already in deps). 5-second timeout, 3 retries on `httpx.ConnectError`, no retries on 4xx/5xx.
- Response parsed into `(date, currency, rate)` tuples and inserted with `INSERT OR IGNORE`.

### 5.5 Tests

- Unit: stubbed BoI response (`tests/fixtures/fx/boi_2026-04.json`), assert `rate("USD", "ILS", date(2026, 4, 8)) == 3.6512` (or whatever the fixture contains).
- Cache-hit: second call hits no HTTP (mock the client, assert call count == 0 on second invocation).
- Weekend walkback: request `date(2026, 4, 4)` (Saturday) → returns Friday's rate.
- Cross-rate: `convert(100, "USD", "EUR", on)` ≈ rate USD→ILS divided by rate EUR→ILS, times 100.
- `FXRateUnavailable`: request date outside cache range and stub HTTP failure.

### 5.6 Out of scope for wave 1

- Currencies beyond what statements actually contain (USD/EUR/GBP/ILS).
- Real-time / intraday rates.
- Manual rate overrides via config.
- Use by parsers (parsers stay FX-naive; conversion is downstream-only).

## 7. Verify-phase exercises

Per Q2 / user sign-off, scope is **B: code re-read + corpus aggregate smoke**. No per-row review.

### 7.1 Code re-read

Files: every file under `argosy/services/expense_ingest/*.py`, `argosy/api/routes/expenses.py`, `argosy/cli/expenses.py`, `argosy/services/fx/*.py`.

Per file, look for:
- Edge cases not covered by tests (empty file, single-row file, all-refund, mixed-currency, regex misfire).
- Error paths: parser failure, FX-unavailable, source-conflict (same external_id but different cardholder), partial-batch LLM failure.
- Concurrency: anything that calls `session.flush()` mid-loop and could land partial state on exception.
- Type safety: pydantic round-trips, especially `amount_nis: float | None` post-migration.
- Filesystem / env injection: anything reading `os.environ` or paths without sanitizing.

Output: a per-file checklist with `(file, finding-summary, severity, in-scope-for-wave-1?)` rows.

### 7.2 Corpus aggregate SQL exercises

Eight queries against the freshly-reingested DB. Each has a written pass/fail interpretation that lands in the verify-findings report.

```sql
-- 1. Source count + statement count per source.
--    Pass: 5 sources (leumi 44745280, isracard 1266, isracard 0235, max 6225, discount 2923),
--          statement counts within ±2 of folder counts.
SELECT src.issuer, src.external_id, COUNT(s.id) AS stmts,
       MIN(s.period_start), MAX(s.period_end)
FROM   expense_sources src LEFT JOIN expense_statements s ON s.source_id = src.id
WHERE  src.user_id = 'ariel'
GROUP  BY 1, 2;

-- 2. Foreign-currency row counts.
--    Pass: > 0 for Isracard (your USD/EUR rows), 0 for Leumi/Max/Discount.
SELECT src.issuer, COUNT(*)
FROM   expense_transactions tx JOIN expense_sources src ON tx.source_id = src.id
WHERE  tx.amount_nis IS NULL AND tx.user_id = 'ariel'
GROUP  BY 1;

-- 3. Bank↔card correlation rate per Max source.
--    Pass: > 80% (post-fix; pre-fix was ~0% because external_id 5280 didn't match the bank's 6225).
SELECT src.external_id,
       SUM(CASE WHEN tx.is_card_payment_match THEN 1 ELSE 0 END) * 1.0 / COUNT(*)
FROM   expense_transactions tx JOIN expense_sources src ON tx.source_id = src.id
WHERE  src.issuer = 'max' AND tx.user_id = 'ariel'
GROUP  BY 1;

-- 4. Per-statement conservation gap.
--    Pass: ABS(gap) < ₪0.50 for every NIS-declared statement.
SELECT id, src.issuer, ABS(s.parsed_total_nis - s.declared_total_nis) AS gap
FROM   expense_statements s JOIN expense_sources src ON s.source_id = src.id
WHERE  s.declared_total_nis IS NOT NULL AND s.user_id = 'ariel'
ORDER  BY gap DESC LIMIT 20;

-- 5. Categories: count by source per category.
--    Pass: no whole-source uncategorization (every source has < 50% rows in 'uncategorized').
SELECT src.issuer, c.name, COUNT(*)
FROM   expense_transactions tx JOIN expense_sources src ON tx.source_id = src.id
       LEFT JOIN expense_categories c ON tx.category_id = c.id
WHERE  tx.user_id = 'ariel'
GROUP  BY 1, 2 ORDER BY 1, 3 DESC;

-- 6. Refund matching rate.
--    Pass: > 60% of refunds matched to a prior debit (rest are statement-edge refunds with no
--          matching debit in the corpus window).
SELECT SUM(CASE WHEN refund_matched_to_id IS NOT NULL THEN 1 ELSE 0 END) AS matched,
       COUNT(*) AS total_refunds
FROM   expense_transactions
WHERE  direction = 'credit' AND tx_type = 'refund' AND user_id = 'ariel';

-- 7. Duplicate detection.
--    Pass: 0 rows. Each (source, date, merchant_normalized, amount_nis) tuple should appear once.
SELECT source_id, occurred_on, merchant_normalized, amount_nis, COUNT(*)
FROM   expense_transactions
WHERE  user_id = 'ariel'
GROUP  BY 1, 2, 3, 4 HAVING COUNT(*) > 1;

-- 8. raw_row JSON shape consistency.
--    Pass: every Leumi tx has 'date' key; no parser produces purely-integer keys post-fix.
SELECT src.issuer, json_extract(tx.raw_row, '$.date') IS NOT NULL AS has_date_key, COUNT(*)
FROM   expense_transactions tx JOIN expense_sources src ON tx.source_id = src.id
WHERE  tx.user_id = 'ariel'
GROUP  BY 1, 2;
```
