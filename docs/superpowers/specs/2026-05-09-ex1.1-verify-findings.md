## New findings from T15 corpus-replay

### Finding 1 — Leumi parser: bidi marks + HTML comments around account number

**Severity:** RED (would break operational backfill on real data).

**Detail:** Real Leumi May 2026 export interleaves Unicode bidi formatting marks (`‏` RTL, `‎` LTR) and HTML comments (`<!---->`) between the Hebrew `חשבון` label and the digits. The synthetic `leumi_osh_minimal.xls` fixture had neither, so `_LEUMI_ACCOUNT_RE` was never exercised against real data and produced empty match → `_leumi_source_hint_assert` raised `ValueError("Leumi account mismatch: expected 44745280, got '')`.

**Fixed in:** commit `2258d39` — strip bidi marks + HTML comments before regex, lazy-match up to 200 chars between label and first digit run, drop English `account` alternation (it false-matched HTML class names like `account-number`). All three Leumi fixtures now extract `44745280` correctly: minimal, real-May-2026, wrong-acct (returns `99999999`).

### Finding 2 — Leumi: SpreadsheetML XML format files unsupported

**Severity:** YELLOW (some Leumi exports won't ingest; user can re-export in HTML format manually).

**Detail:** The user's corpus contains TWO formats of Leumi export:
- HTML-as-xls (the format the parser handles): `2026/Leumi/leumi_2026_May_Osh.xls` works.
- SpreadsheetML XML (older Excel XML): `Resources/Leumi_26_Mar_24.xls` and `Resources/Leumi_26_May_01.xls`. These have no `חשבון` label and can't be parsed by `pd.read_html` → ingest fails per file.

**Status:** **Deferred** — not fixed in EX1.1. Operational impact is "those specific files fail to ingest with a clear error per file"; user can re-export them in HTML format manually or wait for a follow-up parser. Roadmap: add a sniff branch in `services/expense_ingest/sniff.py` that detects SpreadsheetML, then either error with a guide message ("re-export this file in HTML format from Leumi web") OR add a new parser path.

### Finding 3 — Plan-stub defects (multiple tasks)

Four plan-stub defects were discovered by implementers and worked around per-task. Patching them inline in the plan would help future executors:

1. `ExpenseTransaction(statement_id=None)` violates NOT NULL. Workaround: seed real `UserFile` + `ExpenseStatement`. Affected: T7, T8, T13, T15.
2. `CategorizeResult(...)` requires `rationale: str` field. Workaround: add `rationale="stub"`. Affected: T7, T8, T15.
3. `asyncio.run(catalog_upload(...))` inside a sync `Session` causes "database is locked" with aiosqlite. Workaround: direct `UserFile` seed mirroring `tests/test_expense_orchestrator.py::_file()`. Affected: T10, T15.
4. `is_card_payment_match` column doesn't exist; model uses `is_card_payment` (bool) only. Affected: T15.

**Status:** Worked around per-task; documented here for SDD reference.

### Finding 4 — `_USD_NIS_FALLBACK` constant in `parsers/isracard.py` is dead code

**Severity:** INFO (cosmetic).

**Detail:** Line 35 of `parsers/isracard.py` defines `_USD_NIS_FALLBACK = 3.70` with a docstring claiming it's "available for orchestrator-level FX conversion." Post-EX1.1 the FX module (`argosy.services.fx`) supersedes this and the constant is never read. Safe to delete in a future cleanup.

**Status:** Deferred (not bug-blocking).

### Finding 5 — `persistence.py` had 3 NULL-unsafe sites (fixed in T12 commit `baf9c06`)

**Severity:** RED (would crash on first foreign-row insert).

**Detail:** `_content_key` did `f"{tx.amount_nis:.2f}"` (TypeError on None), the dedup query used `Decimal(str(None))` (`InvalidOperation`), and the INSERT call did `Decimal(str(tx.amount_nis))` (same). All three would have crashed the first time the parser emitted a foreign row.

**Status:** Fixed proactively by T12 implementer (commit `baf9c06`); flagged here for visibility.
